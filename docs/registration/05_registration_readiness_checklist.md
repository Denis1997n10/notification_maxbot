# Чек-лист готовности к регистрации ПО

## 1. Статус документа

Документ является рабочим чек-листом подготовки к регистрации программного обеспечения. Он фиксирует, какие сведения уже можно описать на основании текущего кода и архитектуры, а какие должны быть предоставлены правообладателем.

## 2. Что можно подтвердить по текущему проекту

| Блок | Статус | Комментарий |
|---|---:|---|
| Назначение ПО | Частично готово | Описано как платформа подписок и уведомлений жителей по объектам жилищной инфраструктуры. |
| Пользовательская инструкция | Стартовая версия готова | См. `01_user_manual.md`. Требуются финальные скриншоты интерфейса. |
| Функциональные характеристики | Стартовая версия готова | См. `02_functional_characteristics.md`. Нужно сверить с фактическим production scope. |
| Техническое описание | Стартовая версия готова | См. `03_technical_description.md`. Требуется актуализировать после стабилизации production deploy. |
| Инструкция по установке/эксплуатации | Стартовая версия готова | См. `04_installation_and_operation.md`. |
| Архитектурные ограничения | Есть | Зафиксированы в `AGENTS.md`. |
| Dev deployment workflow | Частично готов | Требуется финальная проверка на чистом окружении. |
| Production deployment workflow | Частично готов | Требуются реальные секреты, подтвержденный домен/URL и e2e-проверка. |
| Минимизация данных | Описана | Не хранить квартиры, фото и детальные delivery logs. |

## 3. Что должен предоставить правообладатель

### 3.1. Сведения о правообладателе

- Полное наименование юридического лица или ИП.
- ОГРН/ОГРНИП.
- ИНН.
- Адрес регистрации.
- Контактное лицо.
- Контактный email и телефон.
- Сведения о сайте продукта, если есть.

### 3.2. Подтверждение прав на ПО

Нужно собрать и проверить:

- договоры с разработчиками/подрядчиками;
- акты передачи исключительных прав;
- трудовые договоры и служебные задания, если разработка велась сотрудниками;
- документы о создании ПО;
- документы о владении исходным кодом;
- сведения о репозитории и контроле доступа;
- внутреннее решение о вводе ПО в эксплуатацию или о готовности версии.

### 3.3. Сведения о сторонних компонентах

Нужно подготовить реестр:

- Python-зависимости;
- Node.js-зависимости frontend;
- Terraform provider dependencies;
- open-source лицензии;
- коммерческие компоненты, если используются;
- ограничения лицензий;
- подтверждение, что сторонние компоненты не нарушают права третьих лиц.

Шаблон расположен в `06_components_and_rights_register.md`.

### 3.4. Сведения о технической независимости

Нужно отдельно проверить:

- какие компоненты являются российскими/нероссийскими;
- зависит ли ПО от иностранных облачных сервисов;
- можно ли развернуть продукт в российской инфраструктуре;
- есть ли обязательные внешние SaaS-сервисы;
- как заменить MAX, RegionCity/MPOISK или Yandex Cloud при необходимости.

Текущий проект ориентирован на Yandex Cloud и MAX. Это надо честно указать и не пытаться скрывать.

### 3.5. Экономические сведения

Для подачи могут потребоваться сведения, которые не содержатся в коде:

- модель распространения;
- лицензионная политика;
- сведения о выручке;
- сведения о выплатах по лицензионным договорам;
- сведения о технической поддержке;
- сведения о сопровождении и обновлениях.

## 4. Документы, которые стоит подготовить дополнительно

1. Паспорт программного обеспечения.
2. Описание жизненного цикла разработки и поддержки.
3. Политика информационной безопасности.
4. Модель угроз в упрощенном виде.
5. Регламент управления версиями.
6. Регламент резервного копирования и восстановления.
7. Описание API и интеграций.
8. Документ с архитектурными диаграммами.
9. Скриншоты интерфейса.
10. Инструкция администратора в расширенной версии.

## 5. Риски для регистрации

| Риск | Почему важно | Что сделать |
|---|---|---|
| Не подтверждены исключительные права | Без прав невозможно корректно заявлять ПО | Собрать договоры и акты |
| Нет реестра сторонних компонентов | Нужно понимать лицензии и ограничения | Сформировать SBOM / dependency register |
| Production не стабилизирован | Экспертиза может потребовать демонстрацию | Довести deploy и smoke до повторяемого состояния |
| Документация описывает больше, чем реализовано | Это создает риск отказа или вопросов | Разделять реализовано / запланировано |
| Сильная завязка на конкретные внешние сервисы | Нужно корректно описать зависимости | Указать адаптерную архитектуру и текущие ограничения |
| Нет скриншотов | Пользовательская документация будет неполной | Сделать скриншоты после UI smoke |

## 6. Следующий практический план

1. Стабилизировать dev deploy на чистом окружении.
2. Зафиксировать production deployment workflow без моков.
3. Сделать скриншоты public page, MAX bot flow и admin panel.
4. Обновить `01_user_manual.md` скриншотами и финальными названиями кнопок.
5. Сформировать dependency register из `requirements*.txt` и `package-lock.json`.
6. Подготовить паспорт ПО.
7. Подготовить комплект правоустанавливающих документов.
8. Провести юридическую проверку перед подачей.

## 7. Жесткое замечание

Нельзя пытаться «податься в реестр» только на основании кода и пары markdown-файлов. Реестр — это не только техническая история. Основной риск обычно не в архитектуре, а в правах, зависимостях, лицензионной модели и доказуемости российского происхождения ПО. Если это не закрыть заранее, техническая документация не спасет заявку.
