# Инструкция пользователя

## 1. Назначение системы

Платформа подписок и уведомлений жителей по событиям объектов жилищной инфраструктуры предназначена для информирования пользователей о событиях, связанных с объектами справочника. В текущей версии MVP основной сценарий - уведомление жителей о выполнении работ по уборке подъездов на основании событий, полученных из внешней системы RegionCity/MPOISK.

Система спроектирована как универсальная платформа подписок и уведомлений. В дальнейшем к ней могут подключаться другие типы объектов, внешние источники событий и каналы коммуникации без изменения базовой доменной модели.

## 2. Основные роли пользователей

### 2.1. Житель

Житель использует публичную страницу объекта, адресный miniapp и MAX-бота. Основные действия:

- открыть публичную страницу подъезда по QR-коду или ссылке;
- перейти в MAX-бота по кнопке «Перейти в MAX»;
- выбрать адрес прямо из меню MAX-бота через кнопку «Выбрать адрес»;
- подписаться на уведомления по выбранному подъезду;
- посмотреть список активных адресов;
- открыть карточку конкретного адреса в боте;
- отписаться от конкретного адреса после подтверждения;
- получить уведомление о событии по объекту;
- увидеть раздел будущих услуг как зарезервированную функцию платформы.

### 2.2. Администратор района

Администратор района управляет справочником объектов в пределах назначенных районов. Основные действия:

- войти в административную панель;
- просматривать доступные районы;
- создавать и редактировать справочные объекты в рамках своих прав;
- управлять улицами, домами и подъездами;
- получать публичные ссылки и MAX deep links для подъездов;
- деактивировать объекты вместо физического удаления;
- отправить тестовое уведомление, если доступно в текущем окружении.

### 2.3. Суперадминистратор

Суперадминистратор имеет расширенные права. Основные действия:

- управлять всеми городами, районами, улицами, домами и подъездами;
- назначать районы администраторам;
- управлять административными пользователями;
- просматривать пользователей-жителей;
- деактивировать пользователей и объекты;
- экспортировать адресный справочник;
- выполнять предварительную проверку и импорт адресного справочника;
- выполнять проверочные действия после развертывания системы.

## 3. Работа жителя

### 3.1. Открытие публичной страницы подъезда

1. Пользователь сканирует QR-код, размещенный у подъезда, или переходит по прямой ссылке.
2. Система открывает публичную страницу объекта.
3. На странице отображается информация об объекте: район, дом, подъезд, адрес и доступные действия.
4. Если для объекта есть события, отображается краткая история последних событий. Для MVP целевое значение - последние 10 событий уборки.
5. Пользователь нажимает «Перейти в MAX», чтобы подписаться на уведомления по этому объекту.

### 3.2. Подписка через QR-код или публичную страницу

1. Пользователь открывает публичную страницу подъезда.
2. Нажимает «Перейти в MAX».
3. Ссылка открывает MAX-бота с deep link payload вида `start=e_<public_code>`.
4. Бот получает публичный код объекта и автоматически добавляет адрес в подписки пользователя.
5. Если пользователь уже подписан на этот адрес, повторная подписка не создается.

### 3.3. Подписка через кнопку «Выбрать адрес» в MAX-боте

1. Пользователь открывает MAX-бота.
2. Нажимает кнопку «Выбрать адрес».
3. Открывается адресный miniapp / публичный режим выбора адреса.
4. Пользователь находит поддерживаемый адрес в списке.
5. После выбора адреса система создает подписку на соответствующий подъезд.

Этот сценарий нужен, если пользователь не сканировал QR-код или хочет добавить адрес вручную из меню бота.

### 3.4. Подписка текстовой командой

Если у пользователя есть код из ссылки, он может отправить команду:

`Подписаться КОД`

Также поддерживаются команды вида `/subscribe КОД`, `subscribe КОД` и `+ КОД`.

### 3.5. Просмотр активных адресов

1. Пользователь нажимает «Мои адреса» или отправляет команду `Мои адреса`, `Адреса`, `Подписки` или `/list`.
2. Бот возвращает список активных адресов.
3. Пользователь выбирает адрес кнопкой из списка.
4. Бот открывает карточку адреса и предлагает доступные действия.

### 3.6. Отписка от адреса

1. Пользователь открывает «Мои адреса».
2. Выбирает конкретный адрес.
3. Нажимает «Отписаться от адреса».
4. Бот просит подтвердить действие.
5. После нажатия «Да, отписаться» подписка деактивируется.

Массовая отписка отключена намеренно. Если пользователь отправляет «Отключить все», бот не удаляет все адреса, а предлагает открыть «Мои адреса» и выбрать конкретный адрес. Это снижает риск случайной потери подписок.

### 3.7. Получение уведомлений

1. Внешняя система передает или делает доступным событие по объекту.
2. Платформа сопоставляет событие с объектом справочника.
3. Система находит активные подписки на объект.
4. Пользователи получают уведомление в MAX.

В MVP событием является завершенная уборка подъезда, полученная из RegionCity/MPOISK.

### 3.8. Раздел «Услуги»

Раздел «Услуги» зарезервирован для будущих сервисов платформы. В текущей версии он сообщает пользователю, что сейчас бот отправляет уведомления по выбранным адресам, а дополнительные сервисы будут добавлены позже.

## 4. Работа администратора

### 4.1. Вход в административную панель

1. Администратор открывает URL административной панели.
2. Вводит логин и пароль.
3. Система проверяет учетные данные.
4. При успешной проверке открывается административный интерфейс.

### 4.2. Управление справочником

Администратор работает со справочником объектов:

- город;
- район;
- улица;
- дом;
- подъезд.

Объект «подъезд» является основным объектом подписки в MVP.

### 4.3. Создание подъезда

1. Администратор выбирает город, район, улицу и дом.
2. Открывает список подъездов дома.
3. Создает новый подъезд.
4. Указывает номер подъезда.
5. Указывает внешний идентификатор RegionCity/MPOISK: `regioncity_external_ref`, он же `mapObjectID`.
6. При необходимости указывает публичный код. Если публичный код не указан, система может сгенерировать его автоматически.
7. Система сохраняет подъезд и формирует ссылки:
   - `public_url` - публичная страница подъезда;
   - `max_bot_url` - MAX deep link для подписки на подъезд.

Внешний идентификатор RegionCity/MPOISK обязателен для подъезда. Без него события из RegionCity/MPOISK не смогут быть сопоставлены с адресом и подписчики не получат уведомления.

Система не допускает повторное использование одного и того же `regioncity_external_ref` для разных подъездов. Если идентификатор уже занят, создание или импорт должны завершиться ошибкой конфликта.

### 4.4. Получение внешнего идентификатора объекта RegionCity/MPOISK через интерфейс RegionCity

Для корректного сопоставления событий уборки с подъездом администратор должен указать внешний идентификатор объекта RegionCity/MPOISK. В системе этот идентификатор используется как значение `regioncity_external_ref` подъезда, а во внешних событиях сопоставляется с полем `mapObjectID`.

Порядок получения идентификатора:

1. Открыть интерфейс RegionCity/MPOISK.
2. Перейти на страницу «Объекты»: `https://rc.mpoisk.ru/login/c/Objects/table`.
3. Найти нужный объект через поиск по адресу, например по улице, дому и подъезду.
4. В строке найденного объекта найти ссылку «Активы».
5. Навести курсор на ссылку «Активы», не нажимая на нее.
6. В нижнем левом углу браузера появится URL ссылки. В параметре `objectID` указан нужный идентификатор объекта.
7. Скопировать числовое значение `objectID` и указать его в карточке подъезда как внешний идентификатор RegionCity/MPOISK.

Пример: если в нижнем левом углу браузера отображается ссылка вида `https://rc.mpoisk.ru/login/c/Objects/table?objectID=18950495`, то значение `mapObjectID` / `regioncity_external_ref` для этого подъезда равно `18950495`.

Важно: идентификатор нужно брать именно для того объекта, который соответствует конкретному подъезду. Если выбрать объект дома, улицы или другого подъезда, события из RegionCity/MPOISK не будут сопоставляться с подписками жителей.

### 4.5. Поиск объекта RegionCity/MPOISK из административной панели

В актуальной версии добавлен сценарий поиска объектов RegionCity/MPOISK по адресу из административной панели.

1. Администратор вводит адрес или часть адреса.
2. Система обращается к RegionCity/MPOISK.
3. Возвращаются до 10 наиболее похожих объектов.
4. Для каждого кандидата отображаются:
   - `map_object_id`;
   - адрес;
   - название;
   - тип объекта;
   - оценка совпадения.
5. Администратор выбирает подходящий объект и использует его `map_object_id` как внешний идентификатор подъезда.

Если RegionCity/MPOISK недоступен, система должна показать ошибку доступности внешней системы. В этом случае можно получить идентификатор вручную через страницу «Объекты», как описано в разделе 4.4.

### 4.6. Экспорт адресного справочника

Администратор может выгрузить адресный справочник. Для районного администратора выгрузка ограничивается доступными районами. Суперадминистратор может выгрузить весь справочник.

Выгрузка содержит строки со следующими полями:

- `city` - город;
- `district` - район;
- `street` - улица;
- `house_number` - номер дома;
- `building` - строение или корпус;
- `entrance_number` - номер подъезда;
- `public_code` - публичный код подъезда;
- `regioncity_map_object_id` - внешний идентификатор RegionCity/MPOISK;
- `public_url` - публичная ссылка на страницу подъезда;
- `max_bot_url` - ссылка на MAX-бота для подписки на подъезд.

### 4.7. Импорт адресного справочника

Импорт адресов предназначен для ускоренного наполнения справочника.

Процесс состоит из двух этапов:

1. Предварительная проверка импорта.
2. Применение импорта.

Для каждой строки обязательны поля:

- `city`;
- `district`;
- `street`;
- `house_number`;
- `entrance_number`;
- `regioncity_map_object_id`.

Дополнительные поля:

- `building`;
- `public_code`.

Также система может принять внешний идентификатор в полях `regioncity_external_ref` или `mapObjectID`, но рекомендуемое поле для импорта - `regioncity_map_object_id`.

На этапе предварительной проверки система определяет действие по каждой строке:

- `create` - объект будет создан;
- `update` - существующий объект будет обновлен;
- `error` - строка содержит ошибку и не может быть применена.

Типовые ошибки импорта:

- отсутствует обязательное поле;
- некорректный `public_code`;
- дублируется `public_code` внутри файла;
- дублируется `regioncity_map_object_id` внутри файла;
- `public_code` уже используется другим объектом;
- `regioncity_map_object_id` уже используется другим объектом.

Если хотя бы одна строка содержит ошибку, применение импорта должно быть остановлено до исправления данных.

### 4.8. Деактивация объектов

Физическое удаление объектов не используется как основной сценарий. Вместо этого администратор деактивирует объект. Деактивированный объект скрывается или исключается из активных сценариев, но может оставаться в базе для истории и аудита.

### 4.9. Управление администраторами

Суперадминистратор может создавать административных пользователей, назначать роли и ограничивать доступ районных администраторов назначенными районами.

Требования к административному пользователю:

- роль должна быть `super_admin` или `district_admin`;
- пароль должен быть не короче 12 символов;
- для `district_admin` должен быть назначен хотя бы один район;
- логин администратора должен быть уникальным.

### 4.10. Управление пользователями-жителями

Суперадминистратор может просматривать активных пользователей-жителей и деактивировать пользователя. При деактивации пользователя его активные подписки также отключаются.

### 4.11. Тестовое уведомление

Администратор может отправить тестовое уведомление пользователю, если сценарий доступен в текущем окружении. Для отправки нужен идентификатор пользователя и тестовое описание объекта. Сценарий предназначен для проверки канала уведомлений.

## 5. Ограничения MVP

- Основной объект подписки - подъезд.
- Основной внешний источник событий - RegionCity/MPOISK.
- Основной канал уведомлений - MAX.
- Не хранятся квартиры.
- Не хранятся фотографии работ.
- Детальные логи доставки по каждому пользователю не хранятся.
- Обработка фото реализуется как расширяемая возможность, но не как обязательная часть MVP.
- Сценарий заказа дополнительных услуг зарезервирован архитектурно и может быть скрыт или показан как заглушка.
- Массовая отписка пользователя отключена, чтобы избежать случайного удаления всех адресов.

## 6. Требования к пользователю

Для жителя требуется:

- установленный MAX или доступ к нему;
- возможность перейти по ссылке или QR-коду;
- согласие на взаимодействие с ботом.

Для администратора требуется:

- доступ к административной панели;
- учетная запись администратора;
- актуальные права на город/район;
- современный браузер;
- доступ к RegionCity/MPOISK для получения или проверки `mapObjectID`, если автоматический поиск недоступен.

## 7. Типовые проблемы и действия

### 7.1. Пользователь не видит объект

Проверить, что объект активен, публичный код корректен, QR-код ведет на актуальную публичную страницу, а для подъезда сформированы `public_url` и `max_bot_url`.

### 7.2. Пользователь не получает уведомления

Проверить активность подписки, корректность связанного MAX-пользователя, наличие событий по объекту, доступность канала MAX и корректность `regioncity_external_ref` у подъезда.

### 7.3. Событие не сопоставилось с подъездом

Проверить внешний идентификатор объекта. В MVP поле `mapObjectID` из RegionCity/MPOISK должно сопоставляться с `regioncity_external_ref` подъезда. Если идентификатор неизвестен, его можно найти через поиск RegionCity/MPOISK в административной панели или получить вручную на странице RegionCity/MPOISK «Объекты» через ссылку «Активы», как описано в разделе 4.4.

### 7.4. Администратор не видит район

Проверить роль администратора и назначение района. Районный администратор видит только назначенные районы.

### 7.5. Импорт адресов не применяется

Проверить результат предварительной проверки. Нужно исправить все строки со статусом `error`, особенно обязательные поля, дубли `public_code` и дубли `regioncity_map_object_id`.

### 7.6. Ссылка «Перейти в MAX» не добавляет адрес

Проверить, что в ссылке используется актуальный публичный код подъезда, а MAX deep link сформирован в виде `?start=e_<public_code>`.
