# Функциональные характеристики программного обеспечения

## 1. Назначение

Программное обеспечение предназначено для управления подписками пользователей на объекты справочника, получения событий от внешних учетных систем и отправки уведомлений пользователям через подключаемые каналы коммуникации.

В текущем MVP система используется для уведомления жителей о завершенных работах по уборке подъездов. Архитектура продукта допускает расширение на другие типы объектов, события и каналы уведомлений.

## 2. Основные функциональные подсистемы

### 2.1. Публичная подсистема

Функции:

- отображение публичной страницы объекта по публичному коду;
- отображение сведений об объекте: район, дом, подъезд, адрес;
- предоставление перехода в MAX-бота;
- отображение последних событий по объекту в рамках доступной реализации;
- публичная навигация по городам, районам, улицам, домам и подъездам.

Основные API:

- `GET /api/v1/public/entrances/{publicCode}`;
- `GET /api/v1/public/cities`;
- `GET /api/v1/public/cities/{cityId}/districts`;
- `GET /api/v1/public/districts/{districtId}/streets`;
- `GET /api/v1/public/districts/{districtId}/houses`;
- `GET /api/v1/public/streets/{streetId}/houses`;
- `GET /api/v1/public/houses/{houseId}/entrances`;
- `POST /api/v1/public/miniapp/subscriptions`.

### 2.2. Бот-подсистема

Функции:

- прием входящих webhook-событий от MAX;
- обработка команды запуска;
- привязка пользователя к объекту по публичному коду;
- оформление подписки;
- просмотр подписок;
- отключение подписок;
- ответ на служебные команды;
- заглушка будущих платных или дополнительных услуг.

Основной API:

- `POST /api/v1/bot/webhook`.

### 2.3. Подсистема администрирования

Функции:

- вход администратора;
- получение информации о текущем администраторе;
- управление городами;
- управление районами;
- управление улицами;
- управление домами;
- управление подъездами;
- назначение районов администраторам;
- управление административными пользователями;
- мягкая деактивация объектов и пользователей;
- отправка тестового уведомления.

Основные API:

- `POST /api/v1/admin/auth/login`;
- `GET /api/v1/admin/me`;
- `GET/POST /api/v1/admin/cities`;
- `GET/POST /api/v1/admin/cities/{cityId}/districts`;
- `POST /api/v1/admin/cities/{cityId}/districts/assign`;
- `PATCH /api/v1/admin/cities/{cityId}/deactivate`;
- `GET/POST /api/v1/admin/districts`;
- `GET /api/v1/admin/districts/unassigned`;
- `GET/POST /api/v1/admin/districts/{districtId}/houses`;
- `GET/POST /api/v1/admin/districts/{districtId}/streets`;
- `GET/POST /api/v1/admin/streets/{streetId}/houses`;
- `GET/POST /api/v1/admin/houses/{houseId}/entrances`;
- `PATCH /api/v1/admin/districts/{districtId}/deactivate`;
- `PATCH /api/v1/admin/houses/{houseId}/deactivate`;
- `PATCH /api/v1/admin/streets/{streetId}/deactivate`;
- `PATCH /api/v1/admin/entrances/{entranceId}/deactivate`;
- `GET /api/v1/admin/users`;
- `PATCH /api/v1/admin/users/{userId}/deactivate`;
- `GET/POST /api/v1/admin/admin-users`;
- `PATCH /api/v1/admin/admin-users/{adminId}/deactivate`;
- `POST /api/v1/admin/test-notification`.

### 2.4. Подсистема интеграции с внешними источниками событий

Функции:

- получение событий из RegionCity/MPOISK;
- фильтрация событий по типу и статусу;
- сопоставление внешнего объекта с объектом справочника;
- преобразование внешней структуры события во внутреннюю модель `TaskEvent`;
- предотвращение повторной обработки уже обработанных внешних событий.

Текущие правила MVP:

- `taskID` используется как внешний идентификатор события;
- `taskTypeID=51` соответствует уборке;
- `status=3` соответствует завершенной задаче;
- `mapObjectID` сопоставляется с внешней ссылкой подъезда;
- `lastStatusChangeDate` используется как дата события.

### 2.5. Подсистема уведомлений

Функции:

- формирование полезной нагрузки уведомления;
- выбор шаблона сообщения;
- определение активных подписчиков объекта;
- отправка уведомлений через канал MAX;
- возможность расширения на другие каналы уведомлений.

## 3. Нефункциональные характеристики

### 3.1. Архитектурная расширяемость

Система построена по layered/hexagonal подходу. Доменный и прикладной слои не должны зависеть от YDB, MAX, RegionCity/MPOISK, Yandex Cloud или структуры HTTP/Webhook-событий.

### 3.2. Масштабируемость

Система рассчитана на serverless-развертывание в Yandex Cloud. Основные вычислительные компоненты размещаются в Cloud Functions, справочные и операционные данные — в YDB Serverless, статические интерфейсы — в Object Storage.

### 3.3. Безопасность

- Секреты должны храниться в Yandex Lockbox.
- Токены внешних систем не должны храниться в исходном коде.
- Входящие webhook-события MAX должны проверяться по shared secret, если включена такая проверка.
- Административный интерфейс должен использовать авторизацию.
- Роли администраторов должны ограничивать доступ к объектам.

### 3.4. Минимизация данных

- Квартиры не хранятся.
- Фотографии не хранятся локально.
- Детальные логи доставки по каждому пользователю не хранятся.
- Внешние служебные поля, не предназначенные для жителей, не должны отображаться пользователям.

## 4. Ограничения текущей версии

- Реальный внешний источник MVP — RegionCity/MPOISK.
- Основной канал уведомлений — MAX.
- Timer polling может быть отключен до завершения end-to-end проверки.
- Production-развертывание требует явного подтверждения и настройки реальных секретов.
- Некоторые функции могут быть представлены как заготовки или smoke-сценарии до завершения промышленной настройки.
